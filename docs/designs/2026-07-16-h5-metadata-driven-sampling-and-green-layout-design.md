# H5 元数据驱动的采样率与三光区映射设计

状态：用户已确认（2026-07-16）

## 目标

以每条 H5 数据中的 `frequency` 和 `ppg_config` 作为唯一配置来源，删除基于样本名、窗口长度和信号方差的自动判断，确保训练、验证、测试和部署评估使用完全一致的采样率与三光区映射。

## 元数据规则

- `frequency` 仅允许 `25` 或 `100`。
  - `25`：Stage2 直接在 25 Hz 下提取特征；Stage1 从 25 Hz 降到 5 Hz。
  - `100`：Stage2 先从 100 Hz 降到 25 Hz；Stage1 从 100 Hz 降到 5 Hz。
- `ppg_config` 仅允许 `0`、`1` 或 `2`。
  - `0`：`g1=ch3`，`g2=ch4`，`g3=ch5`。
  - `1`：`g1=(ch3+ch9)/2`，`g2=(ch4+ch10)/2`，`g3=(ch5+ch11)/2`。
  - `2`：`g1=(ch6+ch9+ch12)/3`，`g2=(ch7+ch10+ch13)/3`，`g3=(ch8+ch11+ch14)/3`。

通道编号沿用项目现有零基索引语义。特征池中的 `mode` 名称为兼容现有人工选择 CSV 和模型契约继续保留，但其数值直接等于 `ppg_config`，不再由信号推断。

## 数据流

1. `s01` 扫描 H5 时读取、验证元数据，并写入 `splits.json`。
2. 分组窗口布局优先使用样本父 group 的元数据；若元数据位于窗口子 group，则要求所有窗口值一致。
3. `s02`、`s03` 和 `s06` 只使用扫描得到的 `frequency`、`ppg_config`，不再通过名称、长度或方差推断。
4. PPG、ACC、窗口编号和标签仍在读入排序后统一执行 `[3:-3]`。
5. Stage2 特征仍统一在 25 Hz 下计算，三光区物理顺序保持固定。

## 无效数据处理

以下情况跳过整条数据，并输出 H5 文件、样本名和原因：

- `missing_frequency`
- `invalid_frequency`
- `missing_ppg_config`
- `invalid_ppg_config`
- `inconsistent_metadata`

扫描汇总中分别统计上述原因，不回退到历史自动判断。

## 测试与验收

- 验证 `frequency=25` 不触发 100→25 Hz 降采样，`frequency=100` 保持原降采样行为。
- 验证三种 `ppg_config` 的精确三光区公式。
- 验证缺失、非法和分组内不一致元数据均被跳过并正确计数。
- 验证训练特征中的 `mode` 等于 `ppg_config`。
- 验证 s02、s03、s06 和独立部署契约一致。
- 运行全项目测试、编译检查、错误级静态检查和文档差异检查。
