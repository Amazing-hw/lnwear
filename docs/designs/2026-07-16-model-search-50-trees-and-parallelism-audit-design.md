# 模型搜参 50 棵树上限与并行策略审计设计

状态：用户已确认（2026-07-16）

## 目标

将 XGBoost 模型参数搜索中的 `n_estimators` 统一硬限制为不超过 50，同时确认项目各个适合并行的阶段实际使用受控并行，避免外层并行和模型内部线程嵌套导致资源争抢。

## 树数量约束

- `s05` 默认搜索网格改为 `20,25,30,35,40,45,50`。
- `s08` 下发给 `s05` 的默认搜索网格保持完全一致。
- 所有模型搜索入口解析 `model_search_n_estimators` 后检查最大值；任何候选大于 50 时立即给出明确错误，不静默裁剪。
- 该限制只作用于模型参数搜索。特征筛选、测试夹具或部署验证中固定使用且不超过 50 的小模型保持原逻辑。
- README、CLI 帮助、测试和优化说明同步为最大 50 棵。

## 并行策略

- `s01`：按 H5 文件使用进程池并行扫描，多文件时生效。
- `s02`：按样本使用进程池并行提取 Stage1 窗口，任务量较小时自动串行。
- `s03`：按样本使用进程池并行提取特征，最终按原始索引重组，保证确定性。
- `s04`：按 seed 与 group-fold 任务使用进程池并行；单个模型内部默认单线程。
- `s05`：独立模型候选使用线程池外层并行；每个 XGBoost 候选默认 `n_jobs=1`。
- `s06`：按样本并行推理，legacy 状态机网格也支持并行；worker 内模型强制 `n_jobs=1`。
- `s07`：后处理候选使用进程池并行，但该阶段默认关闭。
- `s08`：统一下发 `n_workers`，模型搜索 worker 默认继承全局 worker，并把 BLAS/OpenMP 线程默认限制为 1。
- 所有阶段解析出的 worker 数不超过实际任务数，避免任务较少时创建空闲进程或线程。
- `WL_FORCE_SERIAL=1` 继续作为 s01–s07 的故障排查回退；小任务自动串行属于避免进程池开销的效率策略，不视为并行失效。

## 验收

- 默认 dry-run 只包含 `20,25,30,35,40,45,50`，不出现 55 或 60。
- `s05` 和 `s08` 对显式大于 50 的搜索候选均拒绝执行并给出清晰原因。
- 并行单元测试确认 worker 继承、候选外层并行、确定性重组和强制串行回退均正常。
- 并行单元测试确认各阶段 worker 数不会超过实际任务数。
- README 的默认网格、并行说明与代码一致。
- 全量测试、编译、生产代码错误级静态检查和差异检查通过。
